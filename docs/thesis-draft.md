# 面向 OI 竞赛的在线评测系统安全考试客户端设计与实现

## 摘要

随着程序设计教学和信息学竞赛训练规模的扩大，在线评测系统已经成为算法教学、作业管理和竞赛组织的重要基础设施。QingdaoU OnlineJudge 具备题库管理、比赛组织、ACM/OI 赛制、远程判题和排名统计等能力，但传统浏览器访问模式难以满足线下机房考试中对切屏、复制粘贴、换机、换 IP 等行为的管控需求。针对这一问题，本文设计并实现了一套面向 OI 竞赛场景的安全考试客户端。系统采用自部署 QingdaoU OnlineJudge 作为服务端，保留其题目、比赛、提交和远程隐藏数据判题能力；客户端采用桌面软件形式，主窗口直接加载自部署 OJ 页面，使学生获得与原 OJ 一致的使用体验，同时自动打开本地 C++ 编辑器和样例运行器，支持本地样例调试与远程正式判题分离。客户端默认启用全屏置顶、复制粘贴拦截、剪贴板清空、失焦锁定、黑名单进程检测和教师密码解锁等监考能力，并预留服务端监考审计模型，用于记录考试会话和违规事件。

关键词：在线评测；OI 竞赛；QingdaoU OnlineJudge；考试客户端；本地样例测评；监考审计

## 第 1 章 绪论

### 1.1 研究背景

在线评测系统能够自动编译、运行和判定程序代码，在程序设计课程、算法训练和信息学竞赛中应用广泛。学生通过系统提交代码，系统在隔离环境中运行测试数据并返回结果，教师可以通过排名和提交记录了解学生掌握情况。

在 OI 赛制中，选手通常需要长时间阅读题目、编写代码、运行样例并多次提交。与普通作业不同，正式考试要求环境受控，隐藏测试数据不能泄露，学生也不能随意切换到浏览器、聊天工具、网盘或外部 IDE。仅使用普通网页 OJ 难以约束这些行为，因此需要一个与 OJ 深度结合的桌面考试客户端。

### 1.2 研究意义

本课题的意义主要体现在三个方面。第一，将成熟开源 OJ 与考试客户端结合，避免从零实现评测系统。第二，在保留原 OJ 页面体验的同时提供本地编辑器和样例运行能力，贴合 OI 选手习惯。第三，通过客户端限制和服务端审计提高作弊行为的发现和阻断能力，为线下机房考试提供工程参考。

### 1.3 研究内容

本文主要完成以下工作：

- 基于 Docker Compose 部署 QingdaoU OnlineJudge，初始化超级管理员并支持中文使用环境。
- 设计桌面考试客户端，主窗口直接加载自部署 OJ 页面。
- 实现本地 C++ 编辑器和样例运行器，支持标准输入输出样例测试。
- 实现全屏置顶、复制粘贴拦截、剪贴板清空、失焦锁定、黑名单进程检测等监考功能。
- 设计后端监考审计模型，为考试会话、违规事件和教师解锁提供数据基础。

## 第 2 章 相关技术

### 2.1 QingdaoU OnlineJudge

QingdaoU OnlineJudge 是一套开源在线评测系统，支持题库、比赛、提交、排名和远程判题。其部署方案基于 Docker Compose，评测链路由后端、JudgeServer 和 Judger 组成。系统支持 ACM/OI 赛制，适合作为本课题服务端基础。

### 2.2 Electron 桌面客户端

Electron 基于 Chromium 和 Node.js，适合将 Web 页面封装成桌面软件。本系统利用 Electron 主窗口直接加载自部署 OnlineJudge 页面，从而保证学生看到的是原 OJ 页面，而不是仿制界面。同时客户端使用 Node.js 调用本地 g++ 编译器，实现本地样例运行。

### 2.3 本地 C++ 编译运行

客户端将代码写入临时目录，调用 MinGW/g++ 编译为可执行文件，再将样例输入写入程序标准输入，收集标准输出和标准错误。运行过程设置超时，避免死循环或长时间占用资源。

### 2.4 客户端监考技术

客户端监考主要包括窗口控制、输入控制和行为检测。窗口控制用于全屏和置顶；输入控制用于拦截复制粘贴快捷键和右键菜单；行为检测用于在窗口失焦或发现黑名单进程时锁定考试界面。由于该方案属于用户态客户端监考，不能声称绝对防作弊，但在受控机房环境中具有实际价值。

## 第 3 章 需求分析

### 3.1 学生端需求

学生需要通过客户端登录 OJ，查看比赛和题目，使用本地编辑器编写 C++ 程序，运行题目样例或自定义样例，并将正式代码提交到 OJ 进行远程判题。

### 3.2 教师端需求

教师需要部署 OJ、创建超级管理员、录入题目、创建比赛、设置参赛用户和 IP 限制。考试过程中教师需要处理客户端锁定和违规情况。

### 3.3 监考需求

系统需要默认限制切屏和复制粘贴。学生离开客户端、使用复制粘贴快捷键、打开黑名单软件时，客户端应锁定考试界面，并要求教师输入密码解锁。

### 3.4 非功能需求

系统应尽量复用成熟开源 OJ，降低实现复杂度。正式测试数据只保存在服务端，客户端只用于样例测试，避免泄题。客户端应提供开发调试模式，方便测试和演示。

## 第 4 章 系统设计

### 4.1 总体架构

系统由三部分组成：自部署 OnlineJudge 服务端、Windows 考试客户端、本地编译运行环境。服务端负责题库、比赛、提交和判题；客户端负责受控考试入口、本地编辑和监考；本地 MinGW/g++ 负责样例编译运行。

### 4.2 服务端设计

服务端保留 OnlineJudge 原有功能。部署完成后，教师以超级管理员身份登录系统，创建题目和比赛。监考扩展模块提供 `ExamPolicy`、`ExamSession`、`ProctorEvent` 等模型，后续可接入客户端心跳和违规上报。

### 4.3 客户端界面设计

客户端主窗口直接打开 `serverBaseUrl` 指向的自部署 OJ。这样学生看到的页面与普通浏览器访问 OJ 一致，不需要重新学习界面。客户端启动时自动打开本地编辑器窗口，编辑器内置 C++ 模板，支持样例输入和运行输出。

### 4.4 监考设计

客户端正式模式默认启用考试限制：

- 全屏和置顶，降低学生切换窗口的可能。
- 拦截复制、粘贴、剪切和右键菜单。
- 周期性清空系统剪贴板。
- 失焦后显示锁定窗口。
- 检测浏览器、聊天软件、外部 IDE、远程控制工具等黑名单进程。
- 教师输入解锁密码后恢复考试。

## 第 5 章 系统实现

### 5.1 OJ 从零部署

系统提供 `deploy/bootstrap-onlinejudge.ps1` 脚本，用于克隆官方 `OnlineJudgeDeploy` 的 `2.0` 分支并执行 Docker Compose 启动。部署完成后，使用 `deploy/create-super-admin.ps1` 调用后端容器命令初始化超级管理员。

### 5.2 客户端加载自部署 OJ

客户端配置文件为 `client/oj-shell/gyoj-shell.json`。其中 `serverBaseUrl` 默认设置为 `http://127.0.0.1`，表示本机部署的 OJ。如果部署到云服务器，只需改为对应域名。客户端同时设置 `Accept-Language` 为中文，并尝试向页面写入语言偏好。

### 5.3 本地样例运行

工具窗口中包含 C++ 编辑区、样例输入区、运行按钮和输出区。运行时客户端创建临时目录，写入 `main.cpp`，调用 g++ 编译，成功后运行可执行文件并传入样例输入，最终显示退出码、耗时、标准输出和标准错误。

### 5.4 监考实现

客户端通过 Electron 窗口能力实现全屏、置顶和锁定界面；通过 `before-input-event` 拦截复制粘贴快捷键；通过定时器清空剪贴板；通过 `tasklist` 检测黑名单进程。锁定后显示全屏窗口，只有教师密码正确才能解锁。

## 第 6 章 测试与分析

### 6.1 部署测试

测试内容包括 Docker 是否可用、`OnlineJudgeDeploy` 是否能拉取、容器是否启动、管理员是否能初始化、OJ 页面是否能访问。当前开发机未安装 Docker，因此本机只验证了部署脚本的存在性和命令结构，实际容器测试需在安装 Docker 后完成。

### 6.2 客户端测试

客户端开发模式下验证了主窗口和本地工具窗口可以同时创建。主窗口默认连接自部署 OJ 地址，工具窗口默认打开 C++ 编辑器。正式模式需在考试机房环境中验证全屏、置顶、失焦锁定和剪贴板清理。

### 6.3 本地编译测试

使用简单 A+B 程序进行测试，样例输入 `1 2`，本地 g++ 编译运行后输出 `3`，说明本地编译链路可用。

### 6.4 安全分析

该方案能够提高普通学生在受控机房环境中的作弊成本，但无法对抗内核级工具、虚拟机绕过、系统权限被破坏等高强度攻击。因此论文中应表述为“提高发现和阻断能力”，不应宣称“绝对防作弊”。

## 第 7 章 总结与展望

本文设计并实现了一套基于 QingdaoU OnlineJudge 的 OI 考试客户端原型。系统保留 OnlineJudge 原有判题能力，将本地编辑器和样例运行器集成到桌面客户端，并实现了基础监考限制。后续可以继续完善服务端监考看板、客户端签名、防卸载、自更新、摄像头辅助监考和更完善的网络白名单策略。
